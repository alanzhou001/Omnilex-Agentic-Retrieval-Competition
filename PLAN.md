# Project Implementation Plan

End-to-end plan for the Kaggle **LLM Agentic Legal Information Retrieval** competition. Derived from INTRO.md (competition rules), README.md (repo setup), CLAUDE.md (architecture), and ARCHITECTURE.md (proposed model design).

---

## 1. Goal & Constraints Recap

**Goal**: Given an English legal question, output a semicolon-separated list of Swiss legal citations (federal laws + BGE court decisions, mostly referenced in German) that maximizes **Macro F1** against a hidden gold set.

**Hard constraints**:
- Kaggle notebook, **offline** (no internet, no external APIs)
- ≤ 12h runtime on Kaggle GPU (T4×2 or P100)
- Citations must **exactly match** strings in the retrieval corpus
- Reproducible & scalable (≤ $10 inference cost per sample)

**Available data** (already downloaded):
- `data/llm-agentic-legal-information-retrieval/train.csv` — LEXam training queries + gold
- `data/llm-agentic-legal-information-retrieval/val.csv` — validation queries + gold
- `data/llm-agentic-legal-information-retrieval/test.csv` — test queries only
- `data/llm-agentic-legal-information-retrieval/laws_de.csv` — German federal law corpus
- `data/llm-agentic-legal-information-retrieval/court_considerations.csv` — BGE decisions corpus
- `data/llm-agentic-legal-information-retrieval/sample_submission.csv` — format reference

---

## 2. Solution Overview

Following [ARCHITECTURE.md](ARCHITECTURE.md), the pipeline is:

```
Query (EN)
  → [Optional] EN→DE translation
  → Bi-encoder (multilingual-e5-large, fine-tuned)
  → FAISS ANN search over corpus (top-100)
  → Cross-encoder reranker (mDeBERTa-v3-base, fine-tuned)
  → Score threshold calibrated for Macro F1
  → submission.csv
```

Citations are **never generated by an LLM** — they are always copied from the retrieval corpus (exact-match requirement).

## 3. Phase Plan

### Phase 0 — Environment & Data (Day 1)

**Goal**: verify local + Kaggle environments can load data and run a trivial submission.

Tasks:
1. `pip install -e .` + `pip install -r requirements-dev.txt`
2. Run `pytest` — confirm citation normalizer + metrics tests pass
3. Inspect data shapes + column names in each CSV (`pandas.head()`, `.info()`)
4. Confirm gold citation strings in `train.csv`/`val.csv` are all present in the corpus CSVs (`laws_de.csv` ∪ `court_considerations.csv`). Compute coverage: what fraction of train/val gold citations appear verbatim in the corpus? If <100%, inspect mismatches — this sets the ceiling for any retrieval-based method.
5. Produce a **trivial baseline submission**: predict the top-5 most frequent gold citations from training for every test query. Run `python scripts/evaluate_submission.py` locally against val to establish a floor score.

Deliverables:
- `notebooks/00_data_exploration.ipynb`
- `submission_trivial.csv` (sanity check)
- Coverage report (% of gold found in corpus)

---

### Phase 1 — Strong Baseline: BM25 + Normalization (Day 2-3)

**Goal**: beat the existing notebook baselines with a reliable retrieval-only system, no LLM.

Tasks:
1. Run `python scripts/build_indices.py` to get BM25 indices over `laws_de.csv` + `court_considerations.csv`
2. For each val query:
   - Run BM25 retrieval (top-50 from each corpus)
   - Deduplicate + take top-K by score
   - Emit citations as-is (no LLM normalization)
3. Tune **K** (number of returned citations per query) on val set to maximize Macro F1. Expect an optimum around K=3-10 — too many citations hurts precision.
4. **Cross-lingual probe**: also encode queries with a cheap EN→DE translator (e.g., `Helsinki-NLP/opus-mt-en-de` offline) and merge candidate sets. Measure delta.
5. Save scored run as `submission_bm25.csv`.

Expected Macro F1: ~0.20-0.30

Deliverables:
- `notebooks/03_bm25_tuned.ipynb`
- `submission_bm25.csv`
- Tuning table: `K` vs Macro F1 on val

---

### Phase 2 — Dense Retrieval, Zero-shot (Day 4-5)

**Goal**: establish a zero-shot multilingual dense retrieval baseline before training.

Tasks:
1. Embed the full corpus (~271K docs) with `intfloat/multilingual-e5-large`:
   - Text template: `"passage: {citation} | {title} | {text[:512]}"` (e5 expects `passage:` prefix for corpus, `query:` for queries)
   - Batch size 64, GPU; should take ~30-60 min on T4
   - Save to `data/processed/corpus_embeddings.npy` + `corpus_ids.npy`
2. Build FAISS index:
   - `IndexFlatIP` first (exact, tractable at 271K × 1024 fp32 ≈ 1.1 GB)
   - If slow, switch to `IndexIVFFlat` with nlist=1024
3. Retrieve top-100 per val query, score Macro F1 at different K cutoffs
4. **Hybrid**: combine BM25 + dense scores with reciprocal rank fusion (RRF). Typically beats either alone.

Expected Macro F1: ~0.35-0.45 (zero-shot); ~0.40-0.50 with RRF hybrid.

Deliverables:
- `scripts/embed_corpus.py`
- `notebooks/04_dense_retrieval.ipynb`
- `data/processed/corpus_faiss.index`
- `submission_dense.csv`, `submission_hybrid.csv`

---

### Phase 3 — Fine-Tune Bi-Encoder (Day 6-8)

**Goal**: domain-adapt the bi-encoder on LEXam query/citation pairs.

Tasks:
1. **Build training pairs**:
   - Positives: (query, gold_citation_document_text) from train.csv
   - Hard negatives: top-50 BM25 non-gold results per query (3-5 hardest per positive)
2. **Training**:
   - Framework: `sentence-transformers` (wraps HF correctly for e5)
   - Loss: `MultipleNegativesRankingLoss` (InfoNCE with in-batch + explicit hard negatives)
   - Hyperparams: bs=16-32, lr=2e-5, warmup 10%, cosine decay, 3 epochs
   - Run on local GPU if available; else Colab Pro / Kaggle training notebook
3. Re-embed corpus with fine-tuned encoder, rerun retrieval, measure on val
4. Iterate: if improvement plateaus, try larger batch (via gradient accumulation), longer context, or `BGE-M3` as an alternative backbone

Expected Macro F1: ~0.50-0.60

Deliverables:
- `scripts/train_biencoder.py`
- `models/biencoder/` (fine-tuned checkpoint)
- New corpus embeddings + FAISS index
- `submission_biencoder_ft.csv`

---

### Phase 4 — Cross-Encoder Reranker (Day 9-10)

**Goal**: boost precision by reranking bi-encoder top-100.

Tasks:
1. **Build reranker training pairs**:
   - For each train query, take top-100 from fine-tuned bi-encoder
   - Label: 1 if in gold, 0 otherwise
   - Downsample negatives to ~5:1 negative:positive ratio
2. **Training**:
   - Model: `microsoft/mdeberta-v3-base` (or `BAAI/bge-reranker-v2-m3` if license permits)
   - Input: `"{query} [SEP] {citation} | {title} | {text[:400]}"`
   - Loss: BCE
   - bs=16, lr=1e-5, 2 epochs
3. Inference pipeline:
   - Bi-encoder → top-100 → cross-encoder scores → sort desc
   - Threshold θ: grid search on val to maximize Macro F1
4. **Adaptive K per query**: as a variant, predict the *number* of citations to return (e.g., via simple regression on cross-encoder score distribution features: top-1 score, gap to top-2, etc.). Pick whichever of fixed-θ / adaptive-K does better on val.

Expected Macro F1: ~0.60-0.70

Deliverables:
- `scripts/train_reranker.py`
- `models/reranker/` checkpoint
- `submission_reranked.csv`

---

### Phase 5 — Kaggle Packaging & Submission (Day 11-12)

**Goal**: wrap the trained pipeline in a single offline Kaggle notebook.

Tasks:
1. Upload as **Kaggle datasets**:
   - Fine-tuned bi-encoder weights (~2.2 GB)
   - Fine-tuned reranker weights (~350 MB)
   - Pre-computed corpus FAISS index + corpus_ids.npy (~1.1 GB)
   - `omnilex` library source (zipped) so it can be `pip install`ed offline
2. Create `notebooks/90_submission.ipynb`:
   - Cell 1: `pip install` omnilex + any wheels (offline via Kaggle dataset)
   - Cell 2: load bi-encoder, reranker, FAISS index from `/kaggle/input/...`
   - Cell 3: load `test.csv`
   - Cell 4: for each query → retrieve → rerank → threshold → collect citations
   - Cell 5: write `submission.csv` in required format
3. Verify notebook runs end-to-end with "Internet off" on Kaggle
4. Submit privately first; iterate; submit publicly once stable

Deliverables:
- Kaggle datasets uploaded
- `notebooks/90_submission.ipynb` submitted
- Entry on public leaderboard

### Phase 6 — Ablations & Stretch Goals (if time permits)

Ranked by expected impact:

1. **Query translation** (EN→DE) before dense retrieval. Merge EN + DE retrieval sets.
2. **Citation graph expansion**: if a BGE decision cites an Art. X ZGB, include that statute as candidate. Build graph from corpus text during index-build.
3. **LLM reranker on top-20**: use a small offline GGUF LLM (Qwen2.5-3B or similar) as a final yes/no relevance filter. Slower but can add ~2-5 F1.
4. **Ensemble**: BM25 + bi-encoder + cross-encoder score fusion with learned weights (tiny logistic regression on val features).
5. **ColBERT-style late interaction** if time permits — potentially large recall gain.

## 4. Risk Register

| Risk | Mitigation |
|------|-----------|
| Gold citations don't all appear verbatim in corpus | Phase 0 coverage analysis; if gap is large, look for normalization bugs before model changes |
| Kaggle GPU OOM on e5-large + reranker together | Load models sequentially (bi-encoder done, free GPU, load reranker). Use fp16 |
| 12h runtime exceeded | Pre-compute corpus embeddings offline; only run query encoding + reranking inside the notebook |
| Hidden test distribution shifts from LEXam | Train on full train+val once final arch is chosen; keep val metric as sanity only, do not overfit θ |
| Cross-encoder dominates runtime | Cap candidates at top-50 rather than top-100 if needed |
| Licensing of base models | All suggested models (e5, mDeBERTa, opus-mt) are MIT/Apache/CC-BY — fine |

## 6. Milestones & Success Criteria

| Milestone | Target Macro F1 (val) | Week |
|-----------|-----------------------|------|
| M0 — trivial baseline | any > 0.0 | 1 |
| M1 — BM25 tuned | ≥ 0.25 | 1 |
| M2 — dense zero-shot | ≥ 0.40 | 2 |
| M3 — biencoder fine-tuned | ≥ 0.55 | 2 |
| M4 — full pipeline (bi + cross) | ≥ 0.65 | 2 |
| M5 — submitted on Kaggle | same, offline | 2 |
| M6 — ablations / stretch | ≥ 0.70 | 3+ |

Each milestone gates the next: **do not fine-tune until zero-shot works; do not add a reranker until bi-encoder is solid**. A working simple pipeline beats a broken sophisticated one.

## 7. Working Agreements

- Every retrieval run writes (a) `submission_<name>.csv` and (b) a one-line entry to `runs.md` with: date, script/notebook, val Macro F1, notes. Keeps an audit trail.
- All training scripts must be re-runnable end-to-end with a fixed seed.
- Model checkpoints and embeddings are git-ignored; only code + config + runs.md are committed.
- Kaggle submission notebook is kept minimal — all heavy lifting (training, embedding) happens offline.
